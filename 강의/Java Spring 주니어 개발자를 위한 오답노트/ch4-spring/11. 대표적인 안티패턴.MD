### Smart UI
* 백엔드의 UI는 컨트롤러이다.
* Smart는 비꼬는 말, 컨트롤러는 Smart하면 안된다.
* 컨트롤러는 적절한 서비스를 선택하고 실행하는 역할 정도만 수행해야 한다.
* 사실 OOP에서 컨트롤러 뿐만 아니라 모든 객체는 Smart하면 안된다.
  * 객체는 자신의 역할만을 수행하는 책임을 가지며
  * 자신이 모르는 것을 다른 객체와의 협력으로 처리하는 공동체를 가진다.
* DDD에 의하면 서비스 계층 역시 똑똑하면 안된다.
* 서비스 계층도 도메인 계층에게 작업을 위임하여 비즈니스 로직을 처리하게 된다.

### Transaction Script
* 서비스 계층에 모든 것이 몰려있는 상황
* 서비스에 트랜잭션이 걸리고, 절차지향적인 코드가 스크립트되어 있는 모양에서 따온 말이다.
  * 도메인은 수동적이며
  * 모든 규칙과 지식이 포함되어 있으며
  * 뚱뚱하다.
  * OOP스럽지 않은 상황이다.
  * 테스트도 어렵다. 비즈니스 로직이 서비스에 결부되어 있기 때문에, 여러 어노테이션을 달고 있는 뚱뚱한 테스트를 작성해야 한다.
* 서비스는 작업을 조정하고 비즈니스 로직을 객체에게 위임해야 한다. (그러면 그 객체는 도메인이 된다.)
* 단, 그 어느 객체에도 위임하기 애매한 비즈니스 로직은 새로운 객체에게 할당한다. (Manager, Calculator, ...)
  * 이런 객체는 도메인 서비스라고 부른다.
  * 도메인 서비스가 아니라 도메인이 되어야 하는 것은 아닌지 고민해봐야 한다.
